Order Service — Documentazione Tecnica

1. Responsabilità

Il servizio Order è il punto di ingresso del sistema e l’orchestratore della SAGA. Ha due compiti distinti:

  1. Ricevere richieste HTTP dall’utente e avviare il processo di consegna.
  2. Coordinare i tre step della SAGA reagendo agli eventi AMQP pubblicati dagli altri servizi.

9. AMQP — Gestione ack/nack nel consumer

Perché requeue: false su entrambi gli errori? Un messaggio malformato non diventerà mai valido riprocessandolo. Un errore nell’handler (es. MongoDB non raggiungibile) è invece transiente: in questo caso si potrebbe volere requeue: true, ma si rischia un loop di retry su bug del codice. La scelta conservativa è requeue: false + log dell’errore.

10. Canali AMQP separati

main.rs
├── publish_channel  →  SagaOrchestrator (pubblica eventi e messaggi)
└── consume_channel  →  consumer::start()  (consuma saga_events_queue)

Tenere publish e consume su canali separati evita che la back-pressure del consumer (es. messaggi in accumulo) blocchi la capacità di pubblicare. In lapin, un Channel è una sessione AMQP 0-9-1 multiplexata sulla stessa connessione TCP.

11. API Reference

POST /

Crea un nuovo ordine e avvia la SAGA.

Request body:

{
  "customer_id": "cust-123",
  "from_address": "Via Roma 1, Milano",
  "to_address": "Via Dante 5, Milano",
  "package_weight": 2.5,
  "requested_delivery_time": "2026-04-10T14:00:00Z",
  "max_delivery_time_minutes": 30
}

requested_delivery_time è opzionale: default now + 2h.

Response 201:

{
  "order_id": "550e8400-...",
  "saga_id":  "f47ac10b-...",
  "status":   "SAGA_STARTED"
}

GET /{orderId}/saga-status

Restituisce lo stato corrente della SAGA per un ordine.

Response 200:
