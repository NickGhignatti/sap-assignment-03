Drone Service — Documentazione Tecnica

1. Responsabilità

Il Drone Service è il componente più complesso del sistema. Ha quattro responsabilità distinte:

  1. Gestione droni — riceve ordini e avvia la consegna assegnando un drone.
  2. Event Sourcing — persiste ogni cambio di stato del drone come evento immutabile su MongoDB; lo stato corrente è sempre ricostruibile dalla sequenza degli eventi.
  3. Scheduler — un task periodico (ogni 10 s) controlla se i droni in volo hanno raggiunto la destinazione e aggiorna l’event store di conseguenza.
  4. Compensazione — riceve eventi CompensateDrone dalla SAGA e rimuove il drone dalla mappa in-flight.

5. Event Sourcing — Sequenza completa degli eventi

Per ogni drone, la sequenza di eventi su MongoDB è sempre questa:

version=0  DRONE_CREATED    { drone_id, order_id, customer_id, from, to, weight, ... }
version=1  DRONE_DISPATCHED { drone_id, order_id, dispatch_time }
version=2  DRONE_DELIVERED  { drone_id, order_id, delivery_time }
version=3  DRONE_RETURNED   { drone_id, order_id, return_time }

Invariante garantita da rebuild_drone()

stato_ricostruito_da_eventi == stato_osservato_in_flight_map

La funzione applica gli eventi in ordine di version partendo sempre da Created:

9. InFlightMap — Gestione della concorrenza

pub type InFlightMap = Arc<Mutex<HashMap<String, DroneEntry>>>;

Tre task concorrenti accedono alla stessa mappa:

TaskOperazioneFrequenza
Consumer ordiniinsert(order_id, entry)Ad ogni ordine ricevuto
Schedulercollect arrivati + remove(order_id)Ogni 10 secondi
Consumer compensazioneretain(...) rimuove per drone_idAl ricevimento di CompensateDrone
Handler HTTPlock().get(order_id)Ad ogni chiamata GET

Regola fondamentale: il lock non viene mai mantenuto attraverso un await. In settle_arrived() questo è esplicito:

// 1. Acquisisci lock, raccogli dati, rilascia subito
let arrived: Vec<DroneEntry> = {
    let map = self.in_flight.lock().unwrap();
    map.values().filter(...).cloned().collect()
};   // ← lock rilasciato qui

// 2. Operazioni async senza lock
for entry in arrived {
    self.store.save_event(...).await?;  // nessun lock attivo
    self.in_flight.lock().unwrap().remove(...);  // ri-acquisisce solo alla fine
}

10. Scheduler — Sostituto di @Scheduled

In Java Spring, @Scheduled(fixedDelay = 10000) creava un thread dedicato. In Rust il task periodico è un tokio::spawn con tokio::time::interval:

tokio::spawn(async move {
    let mut interval = tokio::time::interval(Duration::from_secs(10));
    loop {
        interval.tick().await;       // aspetta 10s senza bloccare il thread
        scheduler_svc.settle_arrived().await?;
    }
});

interval.tick() è non-bloccante: cede il controllo al runtime Tokio durante l’attesa, che può usare il thread per altri task.

11. API Reference

GET /order/{orderId}/status

Stato in-flight del drone assegnato all’ordine.

Response 200:

Drone a1b2-... InTransit from 'Via Roma 1' to 'Via Dante 5' — 42s elapsed

Response 404: ordine non trovato nella mappa in-flight (non ancora partito o già completato).

GET /drone/{droneId}/events

Storico completo degli eventi di un drone (Event Sourcing log).

Response 200:

[
  { "event_type": "DRONE_CREATED",    "timestamp": "2026-04-10T14:00:00Z", "version": 0 },
  { "event_type": "DRONE_DISPATCHED", "timestamp": "2026-04-10T14:00:01Z", "version": 1 },
  { "event_type": "DRONE_DELIVERED",  "timestamp": "2026-04-10T14:25:01Z", "version": 2 },
  { "event_type": "DRONE_RETURNED",   "timestamp": "2026-04-10T14:25:01Z", "version": 3 }
]

GET /order/{orderId}/events

Tutti gli eventi associati a un ordine specifico.

GET /drone/{droneId}/rebuild

Ricostruisce lo stato del drone applicando il suo event log in ordine. Utile per verificare la coerenza dell’Event Sourcing.

Response 200:

Drone a1b2-... Returning from 'Via Roma 1' to 'Via Dante 5' — 1502s elapsed

GET /health

Response 200: Drone Service is running

12. Graceful shutdown

axum::serve(listener, app)
    .with_graceful_shutdown(async {
        tokio::signal::ctrl_c().await.expect("...");
    })
    .await?;

A differenza dell’Order Service, il Drone Service usa with_graceful_shutdown su Axum, che aspetta il completamento delle richieste HTTP in corso prima di chiudere il server. I consumer AMQP e lo scheduler vengono interrotti quando il processo termina.
