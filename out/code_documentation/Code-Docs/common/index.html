Common — Libreria Condivisa

1. Scopo

common è una libreria Rust che espone i tipi condivisi tra i tre microservizi. Centralizzare questi tipi garantisce che tutti i servizi usino la stessa struttura dati per serializzare e deserializzare i messaggi AMQP, eliminando ogni possibilità di disallineamento.

2. Moduli

common/src/
├── lib.rs          ← re-export pubblico di events e messages
├── events.rs       ← SagaEvent, DroneEvent
└── messages.rs     ← OrderMessage

3. OrderMessage

DTO che viaggia su due code: order_queue (Order → Delivery) e drone_queue (Delivery → Drone).

┌─────────────────────────────────┐
│         OrderMessage            │
├─────────────────────────────────┤
│ + order_id     : String         │
│ + customer_id  : String         │
│ + from_address : String         │
│ + to_address   : String         │
│ + package_weight: f64           │
│ + requested_delivery_time: Utc  │
│ + max_delivery_time_minutes: i32│
└─────────────────────────────────┘

Decisioni di design

  • PartialEq confronta solo order_id — due messaggi con lo stesso ordine sono considerati equivalenti indipendentemente da eventuali campi aggiornati.
  • Il costruttore ::new() usa impl Into<String> per tutti i campi stringa: evita chiamate .to_string() superflue nei call site (zero-cost se si passa già un String).
  • Display formatta una riga leggibile per i log (omette requested_delivery_time per brevità).

4. SagaEvent

Enum tagged union con #[serde(tag = "type")]. Il campo "type" viene inserito automaticamente nel JSON e usato per deserializzare la variante corretta.

Esempio JSON serializzato

{
  "type": "DELIVERY_SCHEDULED",
  "saga_id": "",
  "order_id": "a1b2-...",
  "delivery_id": "c3d4-...",
  "timestamp": "2026-04-10T14:00:00Z"
}

Helper methods

MetodoTipo di ritornoDescrizione
order_id()&strEstrae order_id da qualsiasi variante
saga_id()&strEstrae saga_id da qualsiasi variante

Entrambi usano pattern matching esaustivo — il compilatore garantisce che nessuna variante venga dimenticata.

5. DroneEvent

Enum degli eventi del ciclo di vita di un drone. Implementa Event Sourcing: ogni istanza è un fatto immutabile persistito su MongoDB.

Campo version

Ogni variante porta un campo version: u64 che rappresenta la posizione dell’evento nel log del drone. Viene calcolato con DroneEventStore::count_events_for_drone() prima di ogni insert, garantendo l’ordinamento cronologico dei documenti MongoDB indipendentemente dall’ordine di arrivo.

Helper methods

MetodoTipo di ritornoDescrizione
drone_id()&strID del drone
order_id()&strID dell’ordine associato
version()u64Posizione nell’event log
event_type()&'static strStringa descrittiva (es. "DRONE_CREATED")
timestamp()DateTime<Utc>Momento di creazione dell’evento

Perché due campi temporali su alcuni eventi?

Dispatched, Delivered e Returned hanno sia timestamp (quando l’evento è stato creato nel sistema) sia un campo semantico (dispatch_time, delivery_time, return_time). Il timestamp è il momento di scrittura nel log; il campo semantico è il momento in cui l’azione si è verificata. In questo progetto coincidono, ma la separazione è utile in sistemi dove gli eventi possono essere processati in ritardo.

6. Serializzazione — Schema complessivo

┌──────────────┐   serde_json::to_vec   ┌────────────────┐
│  Rust enum   │ ─────────────────────▶ │  JSON bytes    │
│  (variant)   │                        │  + "type" tag  │
└──────────────┘                        └────────────────┘
                                               │
                                         AMQP payload
                                               │
┌──────────────┐  serde_json::from_slice ┌────────────────┐
│  Rust enum   │ ◀───────────────────── │  JSON bytes    │
│  (variant)   │                        └────────────────┘
└──────────────┘

Il discriminante "type" permette a serde di sapere quale variante istanziare senza ambiguità, anche se le varianti hanno campi diversi.
