Common — Libreria Condivisa

1. Scopo

common è una libreria Rust (una crate senza binario) che espone i tipi condivisi tra i tre microservizi. È il contratto del sistema: ogni messaggio che viaggia su Kafka è una di queste strutture, serializzata in JSON.

Il motivo per cui questi tipi vivono in un’unica libreria, invece di essere duplicati in ogni servizio, è la sicurezza a tempo di compilazione. Se l’Order Service producesse un OrderMessage e il Delivery Service ne avesse una definizione anche solo leggermente diversa, la deserializzazione fallirebbe a runtime, in produzione, in modo silenzioso e difficile da diagnosticare. Centralizzando i tipi, il compilatore garantisce che produttore e consumatore parlino esattamente la stessa lingua: un cambiamento al formato di un messaggio non compila finché tutti i servizi non sono aggiornati.

// common/src/lib.rs — l'intera superficie pubblica della crate
pub mod events;
pub mod messages;

pub use events::{DroneEvent, SagaEvent};
pub use messages::OrderMessage;

Il lib.rs si limita a dichiarare i due moduli e a ri-esportarne i tre tipi principali. Grazie al pub use, i servizi scrivono use common::OrderMessage invece di use common::messages::OrderMessage: il percorso d’import è più corto e indipendente dall’organizzazione interna dei moduli.

common/src/
├── lib.rs          ← re-export pubblico di events e messages
├── events.rs       ← SagaEvent, DroneEvent (+ test di serializzazione)
└── messages.rs     ← OrderMessage

2. OrderMessage — il messaggio d’ordine

OrderMessage è il DTO (Data Transfer Object) che descrive cosa va consegnato e dove. Viaggia su due topic: order-requests (Order → Delivery) e drone-requests (Delivery → Drone). È sempre lo stesso ordine che attraversa la pipeline.

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct OrderMessage {
    pub order_id: String,
    pub customer_id: String,
    pub from_address: String,
    pub to_address: String,
    pub package_weight: f64,
    pub requested_delivery_time: DateTime<Utc>,
    pub max_delivery_time_minutes: i32,
}
CampoTipoSignificato
order_idStringUUID dell’ordine — chiave di correlazione di tutto il sistema
customer_idStringIdentificativo del cliente
from_addressStringIndirizzo di partenza
to_addressStringIndirizzo di destinazione
package_weightf64Peso del pacco (kg) — confrontato con MAX_PAYLOAD_WEIGHT dall’agente
requested_delivery_timeDateTime<Utc>Orario di consegna richiesto
max_delivery_time_minutesi32Finestra massima di consegna (minuti) — usata per il vincolo di deadline

Decisioni di design

Costruttore ::new() con impl Into<String>. Tutti i campi stringa del costruttore accettano impl Into<String>:

pub fn new(
    order_id: impl Into<String>,
    customer_id: impl Into<String>,
    // ...
) -> Self

Questo permette di chiamare OrderMessage::new("abc", "cust-1", ...) passando direttamente literal &str oppure String già posseduti, senza .to_string() ridondanti. La conversione è zero-cost quando si passa già un String (Rust la ottimizza a un semplice spostamento di proprietà).

PartialEq basato solo su order_id. L’uguaglianza tra due OrderMessage è definita confrontando esclusivamente l’order_id:

impl PartialEq for OrderMessage {
    fn eq(&self, other: &Self) -> bool {
        self.order_id == other.order_id
    }
}

Due messaggi che si riferiscono allo stesso ordine sono considerati equivalenti anche se altri campi differiscono. Questa scelta è coerente con il dominio: un ordine è la sua identità, non l’insieme dei suoi attributi in un dato istante.

Display per i log. L’implementazione di Display produce una riga leggibile che omette requested_delivery_time per brevità — utile nei log di tracing senza appesantirli con timestamp ridondanti (già presenti nel formato del log stesso).

3. SagaEvent — gli eventi della transazione distribuita

SagaEvent è l’enum che modella ogni evento scambiato sul topic saga-events. È una tagged union serializzata con #[serde(tag = "type")]: nel JSON viene inserito automaticamente un campo "type" che identifica la variante, permettendo la deserializzazione corretta anche se le varianti hanno campi diversi.

#[derive(Serialize, Deserialize, Debug, Clone)]
#[serde(tag = "type")]
pub enum SagaEvent {
    #[serde(rename = "ORDER_SAGA_STARTED")]
    OrderSagaStarted { /* ... */ },
    // ...
}

L’enum ha 12 varianti, raggruppabili per ruolo nel ciclo di vita della SAGA:

Eventi di avanzamento (successo)

Variantetype JSONPubblicato daSignificato
OrderSagaStartedORDER_SAGA_STARTED—Saga avviata (evento di audit; porta tutti i campi dell’ordine)
OrderValidatedORDER_VALIDATED—Step 1 superato (audit)
DeliveryScheduledDELIVERY_SCHEDULEDDeliveryStep 2 completato; porta delivery_id
DroneAssignedDRONE_ASSIGNEDDroneStep 3 completato; porta drone_id
OrderCompletedORDER_COMPLETEDOrderSAGA conclusa con successo

Eventi di fallimento

OrderValidationFailedORDER_VALIDATION_FAILEDOrderStep 1 fallito; porta reason
DeliverySchedulingFailedDELIVERY_SCHEDULING_FAILED(Delivery)Step 2 fallito; porta reason
DroneAssignmentFailedDRONE_ASSIGNMENT_FAILEDDroneStep 3 fallito; porta reason

Eventi di compensazione

Variantetype JSONPubblicato daConsumato daCampi aggiuntivi

Tutte le varianti condividono i campi saga_id, order_id e timestamp; le varianti di fallimento e compensazione aggiungono reason; alcune portano identificativi specifici (delivery_id, drone_id).

Esempio JSON serializzato

{
  "type": "DELIVERY_SCHEDULED",
  "saga_id": "",
  "order_id": "a1b2-...",
  "delivery_id": "c3d4-...",
  "timestamp": "2026-04-10T14:00:00Z"
}

Il campo "type" è inserito da serde grazie a #[serde(tag = "type")], e il suo valore deriva dall’attributo #[serde(rename = "...")] di ciascuna variante (es. DELIVERY_SCHEDULED invece del nome Rust DeliveryScheduled). In deserializzazione, serde legge prima "type", poi sa quale variante istanziare e quali campi aspettarsi.

Metodi helper

Estrarre order_id o saga_id da un SagaEvent richiederebbe un match su tutte le 12 varianti a ogni call site. Per evitarlo, l’enum offre due accessor:

MetodoRitornoDescrizione
order_id()&strEstrae order_id da qualsiasi variante
saga_id()&strEstrae saga_id da qualsiasi variante

Entrambi usano un match esaustivo che sfrutta l’or-pattern di Rust (A { order_id, .. } | B { order_id, .. } | ...) per legare il campo comune in un’unica braccia. L’esaustività è garantita dal compilatore: se si aggiungesse una nuova variante senza aggiornare questi metodi, il codice non compilerebbe. È così che l’orchestratore, ricevendo un evento qualsiasi, ne ricava l’order_id per correlarlo alla saga giusta.

4. DroneEvent — gli eventi di Event Sourcing del drone

DroneEvent modella gli eventi del ciclo di vita di un drone. A differenza di SagaEvent (che è effimero, un messaggio di coordinamento), ogni DroneEvent è un fatto immutabile persistito su MongoDB: insieme, gli eventi di un drone costituiscono il suo event log, da cui lo stato corrente è sempre ricostruibile (pattern Event Sourcing — vedi Drone Service.

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(tag = "type")]
pub enum DroneEvent {
    Created    { /* tutti i campi dell'ordine + version */ },
    Dispatched { drone_id, order_id, dispatch_time, timestamp, version },
    Delivered  { drone_id, order_id, delivery_time,  timestamp, version },
    Returned   { drone_id, order_id, return_time,    timestamp, version },
}

Le quattro varianti rappresentano le quattro tappe di una consegna, nell’ordine in cui avvengono. Created è speciale: è l’unica a portare tutti i dati dell’ordine (indirizzi, peso, cliente…), perché è l’evento genesi da cui si ricostruisce lo stato; le altre tre portano solo gli identificativi e il timestamp semantico della loro azione.

Il campo version

Ogni variante porta un campo version: u64 che rappresenta la posizione dell’evento nel log del drone (0, 1, 2, 3…). Viene calcolato chiamando DroneEventStore::count_events_for_drone() prima di ogni inserimento: il nuovo evento prende come versione il numero di eventi già presenti. Questo garantisce un ordinamento cronologico stabile dei documenti su MongoDB, indipendente dall’ordine fisico di scrittura, e permette di ricostruire lo stato applicando gli eventi ordinati per version crescente.

Metodi helper

MetodoRitornoDescrizione
drone_id()&strID del drone
order_id()&strID dell’ordine associato
version()u64Posizione nell’event log
event_type()&'static strStringa descrittiva (es. "DRONE_CREATED")
timestamp()DateTime<Utc>Momento di creazione dell’evento

Come per SagaEvent, questi accessor usano match esaustivi con or-pattern, così il codice del Drone Service può lavorare su un DroneEvent generico senza distinguere la variante quando non serve (per esempio per ordinare o loggare).

Perché due campi temporali su alcuni eventi?

Dispatched, Delivered e Returned hanno sia un timestamp sia un campo semantico (dispatch_time, delivery_time, return_time). La distinzione è concettuale: il timestamp è il momento in cui l’evento è stato scritto nel log; il campo semantico è il momento in cui l’azione si è verificata nel dominio. In questo progetto i due coincidono (vengono entrambi impostati a Utc::now() al momento della scrittura), ma la separazione è una buona pratica per sistemi in cui gli eventi possono essere registrati in ritardo rispetto al fatto che descrivono.

5. Serializzazione — schema complessivo

Tutti e tre i tipi attraversano la stessa pipeline: da struttura Rust a byte JSON in produzione, e viceversa in consumo.

┌──────────────┐   serde_json::to_vec   ┌────────────────┐
│  Rust struct │ ─────────────────────▶ │  JSON bytes    │
│  o variant   │                        │  (+ "type" tag │
│              │                        │   per gli enum)│
└──────────────┘                        └────────────────┘
                                               │
                                       Kafka message payload
                                               │
┌──────────────┐ serde_json::from_slice ┌────────────────┐
│  Rust struct │ ◀───────────────────── │  JSON bytes    │
│  o variant   │                        └────────────────┘
└──────────────┘

Per gli enum (SagaEvent, DroneEvent) il discriminante "type" permette a serde di sapere quale variante istanziare senza ambiguità. Per la struct OrderMessage non serve alcun tag: la forma è fissa.

I test come specifica eseguibile

Il modulo events.rs include due test che documentano e verificano il contratto di serializzazione:

#[test]
fn saga_event_serializes_with_type_field() {
    let event = SagaEvent::OrderValidated { /* ... */ };
    let json = serde_json::to_string(&event).unwrap();
    assert!(json.contains("\"type\":\"ORDER_VALIDATED\""));
    assert!(json.contains("\"order_id\":\"o1\""));
}

#[test]
fn drone_event_round_trip() {
    let event = DroneEvent::Created { /* ... */ };
    let json = serde_json::to_string(&event).unwrap();
    let decoded: DroneEvent = serde_json::from_str(&json).unwrap();
    assert_eq!(decoded.drone_id(), "d1");
    assert_eq!(decoded.event_type(), "DRONE_CREATED");
}

Il primo verifica che il campo "type" venga effettivamente emesso con il nome rinominato (ORDER_VALIDATED). Il secondo verifica il round-trip: serializzare e poi deserializzare restituisce un valore equivalente, con la variante corretta. Questi test sono la garanzia minima che il contratto resti stabile nel tempo.
